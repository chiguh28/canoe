# CANoe 自動テストツール 要件定義書

**Version:** 1.0　|　**作成日:** 2026年3月　|　**ステータス:** 初版

---

## 1. 目的・背景

CANoe を用いた車載通信テストにおいて、テストパターンの手動作成・実行・判定・報告書作成に多大な工数が発生している。
本ツールはこれらの一連の作業を自動化し、テスト工数の削減と品質の均一化を実現することを目的とする。

### 解決する課題

- テストパターンの手動作成に時間がかかる（信号情報の確認・記述の繰り返し）
- CANoe の手動操作によるヒューマンエラーのリスク
- テスト結果の判定・集計作業の非効率
- テスト報告書フォーマットの不統一・作成工数の大きさ

---

## 2. システム概要

本ツールは以下の4つの主要機能で構成されるデスクトップアプリケーションである。  
CANdb/LDF ファイルから取得した信号情報を活用して日本語でテストパターンを作成し、CANoe COM API を通じてテストを自動実行・判定・帳票出力するまでの一連のフローを統合する。

| No.  | 機能名               | 概要                                                                 | 優先度 |
|------|----------------------|----------------------------------------------------------------------|--------|
| F-01 | テストパターン作成   | CANdb/LDF ファイルを読み込み、信号情報を取得して日本語でテストパターンを作成する機能 | 高     |
| F-02 | テスト自動実行       | 作成したテストパターンを基に CANoe を自動操作してテストを実行する機能               | 高     |
| F-03 | テスト結果判定       | 実行ログと期待値を比較し、各テストパターンに対して OK/NG を自動判定する機能         | 高     |
| F-04 | テスト結果帳票作成   | テスト結果を一覧シートおよび詳細シートとして Excel ファイルに出力する機能           | 中     |

---

## 3. 機能要件

### F-01　テストパターン作成機能

#### 3.1.1 信号情報の取得

CANdb（`.dbc`）または LDF（`.ldf`）ファイルを読み込み、テスト対象の信号情報を自動取得する。

| 項目             | 内容                                                                 |
|------------------|----------------------------------------------------------------------|
| 対応ファイル形式 | CANdb 形式（.dbc）、LDF 形式（.ldf）                                |
| 取得信号情報     | メッセージ ID、信号名、データ型、最小値/最大値、物理単位、送受信ノード情報 |
| 信号選択 UI      | ドロップダウン（選択ボックス）方式。信号名・メッセージ名での絞り込み検索対応 |
| 複数ファイル対応 | 複数の CANdb/LDF ファイルを同時に読み込み、統合して表示              |
| データ保持       | 読み込んだ信号情報はセッション中メモリに保持、再読み込み不要         |

#### 3.1.2 テストパターンの構成要素

1件のテストパターンは以下のフィールドで構成される。

| No. | フィールド名       | 説明                                           | 入力方式             |
|-----|--------------------|------------------------------------------------|----------------------|
| 1   | テストケース ID    | 自動採番（TC-001 形式）                        | 自動生成             |
| 2   | テストケース名     | テスト内容を示す名称                           | 日本語テキスト入力   |
| 3   | 対象信号           | テスト対象の信号名（CANdb/LDF から選択）       | 選択ボックス         |
| 4   | 操作内容           | 信号に対する操作を日本語で記載                 | 日本語テキスト入力   |
| 5   | 期待値             | 期待する信号値または状態を日本語で記載         | 日本語テキスト入力   |
| 6   | 前提条件           | テスト実行前の初期状態を日本語で記載           | 日本語テキスト入力   |
| 7   | 待機時間（ms）     | 操作後の待機時間                               | 数値入力             |
| 8   | 備考               | 補足事項                                       | 日本語テキスト入力（任意） |

#### 3.1.3 日本語による自動生成（一括作成）

「操作内容」および「期待値」の日本語記述を **Azure OpenAI** で解析し、CANoe 操作スクリプトのパラメータに自動変換する。  
「作成ボタン」押下時に全テストパターンを一括でスクリプト化する。

**変換例：**

```
入力（日本語）: 「エンジン回転数を 2000rpm に設定して、スロットル開度が 20% 以上になることを確認する」

変換後:
  対象信号   = EngineSpeed
  送信値     = 2000
  期待信号   = ThrottlePosition
  期待条件   = >= 20
```

- 変換不能な記述は警告を表示し、手動修正を促す
- 変換前にプレビュー表示し、ユーザーが確認・修正してから確定できる

### F-02　テスト自動実行機能

#### 3.2.1 CANoe との連携仕様

| 項目                 | 内容                                                                 |
|----------------------|----------------------------------------------------------------------|
| 連携方式             | CANoe COM API を使用（`win32com.client` / Python）                  |
| 対応 CANoe バージョン | CANoe 10.0 以降（COM API 対応バージョン）                           |
| 測定開始/停止制御    | `CANoe.Measurement.Start()` / `Stop()` にて Python から制御         |
| 信号送信             | `CANoe.Bus.Signal("BusName","MsgName","SignalName").Value = value`  |
| 信号受信/ログ取得    | CAPL / Logging 機能を利用。実行後に BLF または CSV 形式でログ取得  |
| テスト環境設定       | CANoe 構成ファイル（`.cfg`）のパスをツール設定に登録               |

#### 3.2.2 テスト実行フロー

1. テストパターンリストから実行対象を選択（全件 or 個別選択）
2. CANoe 構成ファイル（`.cfg`）を自動ロード・測定開始
3. テストパターンを順次実行：前提条件設定 → 信号送信 → 待機 → ログ取得
4. 全パターン実行完了後、測定停止・ログファイル保存
5. 実行中は進捗バーとステータスをリアルタイム表示
6. 実行中断ボタンによる緊急停止対応

### F-03　テスト結果判定機能

#### 3.3.1 判定ロジック

| 判定種別     | 判定条件                    | 処理内容                                     |
|--------------|-----------------------------|----------------------------------------------|
| 値一致判定   | 実測値 = 期待値             | 完全一致する場合 OK 判定                     |
| 範囲判定     | 最小値 ≤ 実測値 ≤ 最大値    | 指定範囲内に収まる場合 OK 判定               |
| 変化検知判定 | 信号値の変化有無            | 期待する変化（増加/減少/変化なし）を確認     |
| タイムアウト判定 | 待機時間内に信号が変化しない | タイムアウトとして NG 判定                 |
| 複合判定     | 複数条件の AND/OR 組み合わせ | 全条件を満たす場合のみ OK 判定              |

#### 3.3.2 判定結果の管理

- 各テストパターンに対して `OK` / `NG` / `ERROR`（実行異常）の3種類で判定
- 判定結果は JSON ファイルに保存し、帳票作成機能へ引き渡す
- NG・ERROR のケースは判定根拠（期待値・実測値・差異）を詳細ログとして記録

---

### F-04　テスト結果帳票作成機能

#### 3.4.1 帳票構成

| シート           | シート名         | 記載内容                                                                               |
|------------------|------------------|----------------------------------------------------------------------------------------|
| サマリシート     | テスト結果一覧   | TC-ID / テストケース名 / 対象信号 / 期待値 / 実測値 / 判定結果（OK/NG）/ 実行日時      |
| 詳細シート①〜N  | TC-XXX 詳細      | 各テストケースの実行ログ（タイムスタンプ付き）、判定根拠、信号波形データへのリンク     |
| 統計シート       | サマリ統計       | テスト総数 / OK 数 / NG 数 / 合格率 / 実行時間合計                                     |

#### 3.4.2 帳票フォーマット

- 出力形式：Excel（`.xlsx`）
- 「帳票作成ボタン」押下で自動生成、出力先フォルダを選択ダイアログで指定
- サマリシート：OK=緑、NG=赤のセルカラーリングで視認性を確保
- 詳細シートはテストパターン数に応じて動的に生成（TC-001 〜 TC-NNN）
- テスト実行日時・ツールバージョン・CANoe 構成ファイル名をヘッダーに自動記載

## 4. 技術仕様

| カテゴリ         | 技術・ライブラリ                        | 用途                               |
|------------------|-----------------------------------------|------------------------------------|
| 言語             | Python 3.10 以上                        | メインロジック全般                 |
| GUI              | tkinter / PyQt5（要検討）               | デスクトップアプリケーション UI    |
| CANoe 連携       | win32com.client（pywin32）              | CANoe COM API 操作                 |
| DBC パース       | cantools                                | CANdb（.dbc）ファイル解析          |
| LDF パース       | ldfparser                               | LDF ファイル解析                   |
| AI（日本語解析） | **Azure OpenAI Service**                | 日本語テストパターンの構造化変換   |
| 帳票出力         | openpyxl                                | Excel ファイル生成                 |
| 設定管理         | JSON / YAML                             | ツール設定・テストパターン保存     |
| ログ             | Python logging                          | 実行ログ管理                       |

### Azure OpenAI 連携仕様

| 項目                 | 内容                                                            |
|----------------------|-----------------------------------------------------------------|
| 使用サービス         | Azure OpenAI Service                                            |
| 使用モデル           | GPT-4o（または GPT-4 Turbo）                                   |
| 呼び出し方式         | REST API（`openai` Python ライブラリ `AzureOpenAI` クライアント使用） |
| 認証方式             | Azure API Key または Microsoft Entra ID（旧 Azure AD）認証     |
| 設定項目             | `AZURE_OPENAI_ENDPOINT` / `AZURE_OPENAI_API_KEY` / `AZURE_OPENAI_DEPLOYMENT_NAME` |
| プロンプト設計       | システムプロンプトで変換ルール・信号リストを渡し、JSON 形式で構造化出力させる |
| エラーハンドリング   | APIエラー・タイムアウト時はリトライ（最大3回）後にユーザーへ通知 |

**実装イメージ：**

```python
from openai import AzureOpenAI

client = AzureOpenAI(
    azure_endpoint=os.getenv("AZURE_OPENAI_ENDPOINT"),
    api_key=os.getenv("AZURE_OPENAI_API_KEY"),
    api_version="2024-02-01",
)

response = client.chat.completions.create(
    model=os.getenv("AZURE_OPENAI_DEPLOYMENT_NAME"),  # デプロイ名
    messages=[
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user", "content": japanese_test_description},
    ],
    response_format={"type": "json_object"},
)
```

## 7. 前提条件・制約事項

### 前提条件

- 実行環境に CANoe がインストールされていること
- CANoe の COM サーバーが有効化されていること
- Python 実行環境（3.10 以上）が構築済みであること
- Azure OpenAI Service のリソースが作成済みで、GPT-4o（または GPT-4 Turbo）のデプロイが完了していること
- Azure API Key または Entra ID 認証情報が取得済みであること

### 制約事項

- 本ツールは Windows 環境専用（CANoe の動作環境に依存）
- CANoe のバージョンにより COM API の仕様が異なる場合があるため、対応バージョンでの動作確認が必要
- Azure OpenAI の日本語解析精度はモデルおよびプロンプト設計に依存するため、変換結果の検証プロセスを設ける
- Azure OpenAI API の利用にはインターネット接続が必要（オフライン環境では利用不可）
- API 利用コストが発生するため、呼び出し回数の最適化（バッチ処理・キャッシュ活用）を実装する
